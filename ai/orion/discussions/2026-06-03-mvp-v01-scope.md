# Discussion — Orion MVP v0.1 scope, stack, acceptance test, UI/UX

**Date:** 2026-06-03
**Mode:** discussion (answers-only)
**Outcome:** MVP v0.1 defined as a local-only GPS track recorder for the Play Store, with reliable screen-off recording as the hard acceptance gate.

## MVP v0.1 — the true first MVP

A record-and-review tracker, shipped to **Google Play** for a known group of willing testers. **Not** built on the `track` POC — `track` is *inspiration only* (it was buggy). Clean rebuild.

**In scope:**
- **Record** a GPS track — start / stop / pause toggle, **background recording (screen off)**
- **Store** locally
- **View stats** — distance, duration, avg/max speed, elevation gain/loss
- **View on map** — track polyline over online base-map tiles
- **List & toggle** saved tracks

**Out of scope (v0.1):**
- GPX import/export
- Accounts / login / cloud sync — **everything local, no login system yet**
- Offline tile downloads (online base-map tiles only)
- Routing, waypoints, sharing/P2P

## Acceptance test (MVP gate)

**A 2-hour walk with the screen off and the app backgrounded produces a continuous, gap-free track** — no dropped segments, accurate distance, surviving Android Doze / battery optimization on a real device. If this passes, v0.1 ships; if not, nothing else matters.

- **Spike this FIRST**, before building stats/UI.
- Android specifics: foreground service + persistent notification, `ACCESS_BACKGROUND_LOCATION`, surviving Doze.
- `track` never tested screen-off recording → unvalidated unknown, likely where it got buggy.

## Test device

- Primary: **Asus Zenfone 10**, Android 15 (shipped Android 13 + 2 promised OS upgrades → 14, 15; plausible by mid-2026).
- Caveat: Asus is close to stock Android = **lenient** background management. Passing here does NOT guarantee aggressive OEMs (Samsung/Xiaomi/Huawei). Validate on an aggressive-OEM phone before public release. (Open: which OEMs do the testers use?)

## Stack — confirmed (still the best choice)

- **Flutter** — keep. Cross-platform (free iOS later), strong plugins, existing experience. Native Kotlin gives marginally better background-location control but is Android-only; not worth it.
- **MapLibre** — keep, for the offline-vector-tiles future. (`flutter_map` raster is simpler for a polyline-only v0.1, but MapLibre avoids a later migration.)
- **Local storage: SQLite** — keep; consider **Drift** (type-safe) over raw `sqflite` for high-volume track points.
- Real risk is **background GPS reliability**, which is stack-agnostic — pick a battle-tested location plugin and budget time there.

## UI/UX — initial proposal (not finalized)

Three screens, record-centric:
1. **Map (home)** — full-screen map, live location, one big **Record** button (→ Pause/Stop while active), compact live-stats strip (time/distance/speed) during recording.
2. **Tracks list** — saved tracks (name, date, distance, duration); tap to open.
3. **Track detail** — polyline on map + full stats; rename/delete; visibility toggle.

Navigation: simple bottom bar (Map / Tracks) or map-home with a list button.
