# Docker UI — Design

PRD: [`prd.md`](prd.md) (`pc-docker-ui`). Parent: [`../design.md`](../design.md)
(`pc-devops`). Worktree: `~/git/worktrees/putcafe/docker-ui`
(branch `feature/docker-ui`).

## Tool choice: Dozzle (not Portainer)

**Dozzle** (`amir20/dozzle:v8`) fits the PRD best: single tiny container, live
logs + per-container CPU/RAM, containers grouped by compose project, container
detail (image, status, ports), and start/stop/restart via
`DOZZLE_ENABLE_ACTIONS=true` — covering PRD §2 and the "secondary" §3 in one
tool. Portainer rejected: full management surface (stacks, images, deploys)
duplicates what CI owns (PRD non-goal) and is a bigger attack surface.
Host CPU/RAM/disk (PRD nice-to-have): dropped for v1.

**Caddy visibility (PRD §2 last bullet):** implicit — the ops page is served
*through* the shared Caddy, so the page loading at all proves Caddy is up.
No separate probe in v1.

## Topology

```
ops.{$ORION_HOST}            new host block in site.caddy → 127.0.0.1:8103
/root/putcafe/ops-ui/        compose stack (project "ops-ui")
  compose.yml                dozzle: socket + ./data:/data, 127.0.0.1:8103→8080
  creds.env                  OPS_UI_USER/OPS_UI_PASSWORD (root 0600, never in git)
  data/users.yml             bcrypt, generated on-box by `dozzle generate`
```

- Port **8103** (next free in putcafe's 81xx range; 8101/8102 = api). Bound to
  127.0.0.1, public access only via Caddy → HTTPS + auto Let's Encrypt cert
  for `ops.46-250-232-224.sslip.io` (sslip.io resolves any `*.<ip>.sslip.io`).
- The UI is VPS-wide (sees all containers) but lives in putcafe's edge config —
  same caveat as the parent design's import line; upstream to Orion later if
  it ever feels wrong.
- Auth = Dozzle's `simple` provider (login form + JWT), NOT Caddy basic_auth:
  keeps `site.caddy` static/committed with zero secrets in git, and the hash
  is generated by the matching dozzle version on the box.
- Docker socket mounted rw (needed for actions). Root-equivalent for anyone
  past the login — accepted: single bcrypt-protected user, HTTPS only,
  no published port.

## Scripts (extend the existing two-layer pattern)

```
scripts/dev/remote/edge/ops-ui/compose.yml   # committed stack definition
scripts/dev/remote/edge/setup-ops-ui.sh      # ON VPS: idempotent provisioner
scripts/dev/local/edge/setup-ops-ui.sh       # laptop orchestrator
```

**Local `setup-ops-ui.sh`:** source `_conn.sh`; create
`.secrets/ops-ui.env` if missing (user `areg`, `openssl rand` password);
upload `site.caddy`+`setup.sh`+`ops.sh` and run remote `setup.sh` (existing
script — wires/validates/reloads Caddy, already idempotent); upload
`ops-ui/compose.yml` + `setup-ops-ui.sh` + a 0600 `creds.env` built from
`.secrets/ops-ui.env`; run remote `setup-ops-ui.sh`; print URL + creds path.

**Remote `setup-ops-ui.sh`:** require docker (die: "run deploy-api.sh first" —
docker is already on the box) + compose.yml + creds.env; generate `users.yml`
to a temp file via `docker run --rm amir20/dozzle:v8 generate`; install only
on change (cmp) → restart dozzle only then; `docker compose up -d` (no build,
pull-on-first-run). Never touches the `api` stack, Orion, or Caddy.

**`ops.sh` (both layers):** `status` gains the ops-ui container state;
`health` also curls `https://ops.<host>/` (expect HTTP 200 = login page).

## Open questions resolved during implementation

- Exact `dozzle generate` output format — trust the command, don't hand-write
  the yaml.
- Whether Dozzle picks up a changed `users.yml` without restart — assume not;
  restart-on-change is unconditional.
